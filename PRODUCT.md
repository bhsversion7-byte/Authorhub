# AuthorHub Product Memo

这份文档给后续维护 AuthorHub 的 agent / developer 使用。它记录产品方向、设计偏好和不能轻易破坏的体验边界。

## 产品一句话

AuthorHub 是写作者用来整理小说宇宙的私人工作区：把多本小说、大纲、设定、人物关系、时间线、主题标签、图片和分享协作放在一个安静、可靠、可长期使用的手稿桌面里。

## 目标用户

- 连载小说作者、同人作者、原创作者、世界观设定创作者。
- 需要同时管理多本小说、复杂人物关系和大量设定的人。
- 对隐私敏感，草稿、秘密设定和未公开内容不能被误分享。
- 使用场景偏长期：反复增删、修改、回看，而不是一次性填表。

## 产品目标

AuthorHub 的目标不是成为通用文档工具，而是成为“小说结构管理工具”。

好的状态是：用户打开页面后，能快速知道自己正在写哪本小说、主要设定是什么、人物之间怎么连、时间线如何排列、哪些内容可以分享给亲友，哪些仍然留在自己的私人工作区。

## 体验原则

1. 稳定优先。保存、删除、分享、撤回、导出都必须可预期，不允许因为 fallback、缓存或自动同步覆盖真实数据。
2. 不打扰写作。动画、纹理和装饰只能服务氛围，不能抢走文字和控件的可读性。
3. 信息密度要温和。桌面端可以紧凑，但不能像企业后台；移动端要避免误触，特别是拖拽排序。
4. 隐私是核心功能。只读分享必须服务端裁剪内容，不能只靠前端隐藏。
5. 设计语言保持一致。纸感、低饱和色、清楚的按钮状态、lucide 图标、柔和阴影和克制动效。

## 视觉方向

- 关键词：手稿纸、温柔、文学感、安静、细致、私人创作桌。
- 色彩：暖白纸面、深咖文字、柔和 Morandi 配色；危险操作使用柔和玫瑰/红色，不用刺眼纯红。
- 动效：轻微、稳定、可预期；拖拽状态要清楚，但不要夸张。
- 避免：冷蓝 SaaS、玻璃拟态过度、黑色重遮罩、AI 味装饰、过多渐变、巨大营销式 hero、卡片套卡片。

## 当前核心功能

- 多小说管理和侧边导航，支持拖拽排序。
- 小说头部元信息：标题、描述、字数、完结时间、类型、发布页。
- 大纲、设定集、主题标签（标签可拖拽排序，标签面板按小说持久化）。
- 专注编辑器（放大/退出/保存的沉浸式编辑弹层）：大纲、设定集、人物背景故事/隐藏设定、时间点发生背景/具体剧情都支持展开为专注编辑，可拆分多个「小标题」子页面。子页面的增删改（结构性变更）立即同步云端，纯打字内容则和表单其它字段一样，走「保存人物/保存时间点」显式保存 —— 这两者曾经被同一个回调混在一起，导致过度同步或不同步都出现过，目前已区分清楚（见 preference.md 2026-07-10）。
- 人物关系星图：节点拖拽、关系线、关系标签、主角关系红线、聚焦和重置；非主角节点按与主角的图距离分环、按连接数微调大小（Obsidian 式聚类）。
- 人物详情：图片、字段、多选标签、秘密/私密备注；切换到另一人物/时间点前，若有未保存修改会弹窗确认，避免静默丢失。
- 多维交互时间线：时间点卡片可手动拖拽排序，AI 交接关键词只取「时间点+事件名」保持简洁。
- 分享弹窗：共同编辑、只读查看、生成/撤回/复制链接；协作者在线状态提示条（不显示自己）；「你离开时协作者编辑了内容」的补看提示。
- 只读分享范围：大纲、设定集、主题标签、星图、人物详情、时间线；服务端裁剪私密字段和外部首发平台链接，防止去匿名化。
- JSON / Markdown 导出。
- 云端保存、本地缓存兜底（登出/注销会清空本地明文缓存）、图片 Storage 迁移（含服务端一次性回填脚本，2026-07-10 已对生产数据实际执行并验证）。
- 响应式适配：手机/平板/桌面/超宽屏（≥2200px 起额外放大排版，Firefox 也有对应 fallback）。

## 数据与安全边界

- Supabase 是生产数据源：Auth、Postgres、RLS、Storage。
- `author_hub_documents.document` 是用户主文档，必须按 `user_id` 隔离。
- 分享相关表必须只暴露必要字段，RPC 要保留权限检查。
- 只读分享不得泄露 `secret`、`hidden`、`privateNote` 等私密字段。
- 如果云端加载失败，本地 fallback/demo 只能用于临时展示，不能覆盖云端文档。
- 任何涉及数据恢复、批量删除、备份回滚的操作，都先只读确认，再让用户明确同意。

## 维护偏好

- 小范围修复优先，不做无关重构。
- 视觉修改必须同时看桌面、手机和 iPad，不破坏当前整体气质。
- CSS 可以继续按功能文件拆分，但避免重复规则和无用旧代码。
- 新增 Supabase schema 时必须配套 migration、RLS、grant 和验证查询。
- 不要提交 `.env`、本地导出稿、用户数据、临时备份、QA 截图或工具缓存。

## 当前高优先级风险

以下是截至 2026-07-10 仍未处理、且已在 preference.md 中有详细记录的真实待办（不是历史遗留的过时条目，遇到旧文档提到"4 本小说疑似被覆盖"之类的表述时，那已经是很早期的一次性排查，不代表当前状态）：

- 备份策略：仍然依赖 Supabase 整库层面的可靠性，没有面向用户的可下载/可恢复 JSON 备份机制。
- Dashboard 仅有设置：Supabase Auth 的 leaked-password protection 尚未开启；`/auth/v1/recover` 等认证邮件仍走 Supabase 内置的 best-effort 发信服务，未配置自定义 SMTP（Resend 中继），偶发 504 超时。
- 权限粒度：分享链接相关表的字段级写权限还需要收紧（细节见 preference.md 的历史记录），属于已知但未修复项。
- 小概率并发问题：`MediaCarousel` 同一毫秒内上传两个同名文件可能产生重复 React key；分享保存冲突走"整份文档最后写入覆盖"而非字段级合并（有专门的用户提示，但不是真正的 CRDT 合并）。
- 移动端误触：拖拽排序必须有明确 handle 或足够阈值，不能影响正常横向滑动。
- 图片存储：客户端的"自愈迁移"（登录时自动把内嵌 base64 图片上传到 Storage）在部分账号上没有触发（原因未深入排查），需要定期跑 `scripts/dry-run-inline-media-migration.mjs` 或 `scripts/migrate-inline-media-to-storage.mjs` 检查是否又有文档膨胀。
